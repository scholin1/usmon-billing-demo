Insurance Billing: legacy and new platform

USMON // Insurance Billing // Architecture

The 18-year-old system, and the platform that replaces it

There was zero documentation for the legacy billing system. We reconstructed its data shape from the source code, read-only, then designed a new one that keeps the meaning and discards the structure. Here is both, side by side.

Proposed: ADR-001 + ADR-002 Synthetic, no real data Legacy read-only, untouched 2026-07-18

What exists today, reconstructed from code

The legacy

A JSP and servlet application on SQL Server, no ORM, no schema documentation. We mined the shape from the SQL buried in the Java. About 28 claim-related tables. The problems are structural, and they are exactly what we do not carry forward.

Shape

One table does four jobs

The wide patient table conflates patient, insured, encounter, and claim header in a single row.

Duplication

Rows cloned per track

That whole row is copied for each billing track, Global / Tech / Pro, linked back by upid.

Logic

Hidden in the database

Business rules live in stored procedures (addClaim4, getNewPointer, fill_data), invisible to the app.

The contract

A flattening view

A patient_ansi view joins ten tables into one row that the 837 exporter reads. Its definition lives only in the DB.

Coupling

Single monolith

Multi-tenant but inseparable; a client cannot be pulled out, and the worklist waits on the old database.

Security

Secrets in source

Credentials and config were kept in source files rather than a managed secret store. Flagged for the platform team; the new design externalizes all of it.

Same meaning, new structure

What changes

The reconstruction told us what every field means. The new model keeps the meanings and fixes the structure.

Legacy, as-isNew platform
One wide table conflates four conceptsSeparate Patient, Insured, Encounter, Claim, each its own table
Row cloned per billing trackOne Encounter, one to three Claims, no demographic duplication
Business logic buried in stored proceduresData-driven rule sets per tenant (Box 33 assignment, scrub, clocks)
Secrets and config kept in sourceAWS Secrets Manager, referenced, never stored in DB or code
Worklist waits on the old monolithOwn indexed Aurora store, synced, so 52K claims feel instant
Single inseparable monolithMulti-tenant and standalone-capable from day one

The new data model

A clean, multi-tenant schema

One Patient and one Encounter, with one to three Claims hanging off it. Normalized, indexed, every table tenant-isolated. Informed by the legacy meanings, deliberately nothing like its structure.

The platform, managed AWS

The stack

Scalable, managed, fast, modern. Every layer is a managed service, no servers to patch, no new EC2.

Database

Aurora PostgreSQL Serverless v2

Managed and autoscaling; row-level tenant isolation; JSONB for flexible config.

Compute

ECS Fargate

Stateless containers, autoscaling, no servers to patch. Honors the no-new-EC2 rule.

Interface

API-first + its own MCP server

The API is the product; every screen is a client. Programmable from outside.

AI

Bedrock + Data Automation

Scrub, coding assist, prefill, document reading. Server-side, PHI stays in AWS.

Identity

Amazon Cognito

One auth plane; federates USMON SSO as an add-on, own user pools standalone.

Events

EventBridge

Carries claim lifecycle events to the Remit and Arbitron modules.

Why this shape

What you asked for

Scalable

Fargate and Aurora autoscale; stateless API. Multi-tenant from day one.

Managed

Every piece is a managed AWS service. Nothing to babysit.

Fast

Own indexed store synced from legacy; the 52K worklist never waits on the old monolith.

Modern

API-first, event-driven, containerized, infra-as-code, AI-native.

Proposed architecture, recorded in ADR-001 (dual-mode) and ADR-002 (platform and stack). The legacy shape was reconstructed by static analysis of the source only; the legacy database was never touched and no real patient data was read or reproduced. All names and tables shown for the new model are design proposals.
