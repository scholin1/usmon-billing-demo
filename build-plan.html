Holding the line

What is in, and what waits

The demo modeled the whole system. Phase 1 ships the front of the pipeline; the rest is Track 2, by addition, not rewrite.

In (Phase 1)

  • The platform spine (Aurora, Cognito, API on Fargate, USMON adapter)
  • Intake worklist at 52K scale
  • Review engine, roles, per-biller default view
  • AI prefill-and-confirm, coding assist, scrub
  • CMS-1500, 837P and paper, Box 33 assignment
  • Clearinghouse manager, basic 835 status ingest
  • Pilot beside legacy

Waits (Track 2)

  • Ongoing/collections worklist, payments, reports
  • Full 835 reconciliation (the Remit module)
  • Arbitron (consumes the events Phase 1 emits)
  • Non-USMON adapters, standalone onboarding
  • Public API and MCP-server hardening
  • Hospital technical-fee invoicing (route to the existing portal, do not rebuild)
  • Write-back beyond claim status (pending Eslam feasibility)

What unblocks the start

Three decisions with you

GATE 1

Bless ADR-001 and ADR-002 so they can be registered upstream.

GATE 2

Pick the pilot client we prove it with.

GATE 3

Is standalone a near-term goal? It decides how much compliance surface we budget now.

After Phase 1

Ongoing and payments, full 835 reconciliation (Remit), Arbitron consuming the IDR and underpayment events, then the standalone track: other-system adapters, self-serve onboarding, and the public API and MCP server hardened for outside customers. That is the become-a-Stedi turn, and the Phase-1 build is shaped so we get there by adding, not rewriting.
